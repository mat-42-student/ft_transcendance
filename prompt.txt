Description du projet :
Site internet permettant de jouer a pong en multijoueur.
Services disponibles :
- gestion utilisateurs
- authentification OAuth
- 2FA
- chat (messages directs)
- gestion d'amis
- tournois
- matchmaking

Contraintes :
Design de l'application en microservices
docker compose
backend en django (endpoints websockets et/ou API REST)
DB postrgesql
vanilla JS cote client

Question :
Y a-t-il une pratique a privilegier entre :
- ouvrir une seule websocket vers un container gateway qui verifiera les JWT, gerera les eventuelles deconnections puis transmettra les requetes aux deep containers pour la plupart des services (sauf le jeu qui pour des raisons de reactivite aura sa propre ws directe ves le container de jeu). Evnetuellement on pourrait transformer certains call api fetch et les faire passer par la websocket qui sera de toute facon ouverte
- le client ouvre une websocket pour chaque service et chacun gere (de)connection et verification des JWT. Il devrait y avoir 4 websockets differentes
